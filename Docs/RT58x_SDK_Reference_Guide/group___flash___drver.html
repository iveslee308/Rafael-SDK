When MCU reboots, the flash controller will operate in 1-bit SPI mode by default. To speed up the read, user can call function flash_enable_qe() to enable controller to operate in 4-bits mode. ◆ flash_read_page()

uint32_t flash_read_page ( uint32_t  buf_addr,
uint32_t  read_page_addr 
)

Read flash page data in nonblocking mode.

Parameters
[out]buf_addrSpecify the address for the read data of the page.
[in]read_page_addrSpecify the page address in flash that required to read.
Returns
Return values
STATUS_SUCCESS— this operation has been executed... but read operation is still processing
STATUS_EBUSY— flash controller is busying, please call this function again when flash finish current operation.

buf_addr must be 4-bytes alignment. One page is 256 bytes, so buf_addr must have 256 bytes for read data, otherwise some data will be overwritten. Please notice that read_page_addr will be ignored the LSB 8 bits, that is LSB 8 bits of address is don't care. This function is NON_BLOCK function. That is user should wait the operation finish (using function, wait flash_check_busy() return 0, User can not call any flash function before this function finish.

Remarks
Using this flash_read_page API DOES NOT through cache controller!

◆ flash_set_status_reg()

uint32_t flash_set_status_reg ( const flash_status_t *  status)

Set flash status register.

Parameters
[in]statusThe pointer of flash_status_t
Returns
Return values
STATUS_EBUSY— previous flash command not finished, please call this function again after flash idle.
STATUS_SUCCESS— this operation has been finished

Flash driver will write status of status.status1, status.status2 status.status3 into flash, depends on status.require_mode. require_mode is bitwise operation indicator, so it can write multiple status byte in one function call. Current flash does NOT have status3 byte, it reserved for future.

Notice: PLEASE read flash datasheet for write status setting information before you write. Some status bitfield is OTP bit, it just can write once and can not erase again!!! This function is BLOCK function.

For current flash, writing status2 needs to write status1 byte, too. So please remember to read status1 byte first, then save this status1 data in status.status1.

◆ flash_verify_page()

uint32_t flash_verify_page ( uint32_t  read_page_addr)

Check the page crc.

Parameters
[in]read_page_addrSpecify the address of the flash to be verified.
Returns
Return values
STATUS_SUCCESSerase opertation start to processing.
STATUS_EBUSYflash controller is busying, please call this function again when flash finish current operation.

This function is asynchronous NON-BLOCKING mode. That is, when this function returns, it does NOT mean the operation finish. It just means the counting crc8 operation is in progressing. ◆ flash_write_page()

uint32_t flash_write_page ( uint32_t  buf_addr,
uint32_t  write_page_addr 
)

Write/Program flash page data.

Parameters
[in]buf_addrSpecify the address of the write data.
[in]write_page_addrSpecify the page address in flash that to be written.
Returns
Return values
STATUS_SUCCESS— this operation has been executed... but write operation is still processing
STATUS_EBUSY— flash controller is busying, please call this function again when flash finish current operation.

buf_addr must be 4-bytes alignment. One page is 256 bytes, so buf_addr must have 256 bytes for write data, if the data you want to write is less than 256 bytes, please write "0xFF" for padding bytes. Please notice that write_page_addr will be ignored the LSB 8 bits, that is LSB 8 bits of address is don't care. This function is NON-BLOCKING function. Please call function flash_check_busy() to check write operation finish or not. Before writing finish, system SHOULD NOT call any other flash operation!

Remarks
Using this flash_write_page API DOES NOT through cache controller! So after write API finish, please call flush_cache() to refresh cache.
